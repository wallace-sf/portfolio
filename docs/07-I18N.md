# 07 — Internationalization (i18n)

> UI translations, domain content localization, and locale routing.

---

## Supported Locales

| Locale | Language | Role |
|--------|----------|------|
| `en-US` | English | `defaultLocale` |
| `pt-BR` | Brazilian Portuguese | Required for all domain content |
| `es` | Spanish | Optional, falls back to `pt-BR` |

Defined in `apps/web/src/i18n/routing.ts`:

```typescript
locales: ['en-US', 'es', 'pt-BR'],
defaultLocale: 'en-US',
```

---

## UI i18n — `next-intl`

**Configuration:**
- Routing: `src/i18n/routing.ts` — `defineRouting`, `createNavigation`
- Server: `src/i18n/request.ts` — `getRequestConfig`, loads `messages/{locale}.json`
- Middleware: `src/middleware.ts` — locale detection and routing

**Usage in components:**

```typescript
import { useTranslations } from 'next-intl';

const t = useTranslations('Home');
t('projects_title');
```

**Adding UI translations:**
1. Edit `messages/{pt-BR,en-US,es}.json`
2. Keep the same keys across files (rely on `defaultLocale` fallback for missing keys)
3. Use `useTranslations('Namespace')` and `t('key')` in components

## Domain Error Messages

Domain errors use **stable codes** (`INVALID_SLUG`, `ERROR_INVALID_ID`, etc.). The edge resolves a localized message:

- **Source**: `ERROR_MESSAGE` in `@repo/core/shared/i18n/` (`pt-BR`, `en-US`)
- **Rule**: the domain never chooses a locale — callers (web / API) resolve translated messages from `ERROR_MESSAGE` using the request locale
- **`es`**: not yet in `ERROR_MESSAGE`; extend when added

**Adding domain error messages:**
1. Add the code to `ERROR_MESSAGE` in `@repo/core` for `pt-BR` and `en-US`
2. Ensure edge-level mapping reads from `ERROR_MESSAGE[locale][code]`

## Domain Content — `LocalizedText`

For multilingual content (project titles, experience descriptions, etc.), the domain uses `LocalizedText`:

```typescript
// pt-BR is required; en-US and es are optional
LocalizedText.create({ 'pt-BR': 'Meu Projeto', 'en-US': 'My Project' })

// Retrieving content for a locale (with fallback)
text.get('en-US', 'pt-BR') // falls back to pt-BR if en-US is absent
```

**Strategy options for persistence:**
- **Option A** — `LocalizedText` VO: `{ 'pt-BR': string, 'en-US': string }`
- **Option B** — locale-specific columns: `title_pt_br`, `title_en_us`
- **Option C** — translation table: `project_translations(project_id, locale, title, ...)`

The production strategy is still being defined (see [03-BOUNDED-CONTEXTS](./03-BOUNDED-CONTEXTS.md)).

---

## Fallback Chain

1. Requested locale → 2. `defaultLocale` (`en-US`) → 3. `pt-BR` → 4. any available value
