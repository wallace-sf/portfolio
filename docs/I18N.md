## Supported Locales

| Locale | Usage |
|--------|-------|
| **pt-BR** | Brazilian Portuguese |
| **en-US** | US English — `defaultLocale` |
| **es** | Spanish |

Defined in `apps/web/src/i18n/routing.ts`:

```ts
locales: ['en-US', 'es', 'pt-BR'],
defaultLocale: 'en-US',
```

## i18n in the UI

### Current State

- The UI already uses `next-intl` in `apps/web`.
- Locale routing and the `messages/*.json` files already represent real project behavior.

- **Library**: [next-intl](https://next-intl-docs.vercel.app)
- **Config**:
  - **Routing**: `src/i18n/routing.ts` — `defineRouting`, `createNavigation` (`Link`, `redirect`, `usePathname`, `useRouter`)
  - **Server**: `src/i18n/request.ts` — `getRequestConfig`; loads `messages/{locale}.json`
- **Middleware**: `src/middleware.ts` — `createMiddleware(routing, { localeDetection: true })`; matcher for `/`, `/(en-US|es|pt-BR)/:path*`, and exclusions such as `api`, `_next`, `_vercel`, and static files
- **Messages**: `messages/pt-BR.json`, `messages/en-US.json`, `messages/es.json`

### Usage in Components

```ts
import { useTranslations } from 'next-intl';

const t = useTranslations('Home');
t('projects_title');
```

## Domain Content

### Errors (Codes + Messages)

- **Core** (`@repo/core`): `ERROR_MESSAGE` in `shared/i18n/ERROR_MESSAGE.ts` — map for `pt-BR` and `en-US` with stable codes such as:
  - `INVALID_DATE_TIME`, `INVALID_ID`, `INVALID_NAME`
- **Rule**: the domain **does not choose locale**; callers (web / API) use the request locale and resolve a translated message from a dictionary such as `ERROR_MESSAGE` or an edge-level translation layer.
- **`es`**: not yet available in `ERROR_MESSAGE`; when added, extend the relevant i18n types in `@repo/utils` and update Core implementations.

### Multilingual Content (Projects, Posts, etc.)

- **Current state**: project and experience texts are mostly single-language or hardcoded. There is no production-ready `LocalizedText` flow or locale-specific database schema yet.
- **Target model**:
  - **Option A — `LocalizedText` (VO)**: values like `{ 'pt-BR': string, 'en-US': string, 'es'?: string }`; the use case or repository resolves the locale-specific text.
  - **Option B — locale-specific columns**: `title_pt`, `title_en`, `title_es` (or `title_pt_br`, etc.); the mapper selects the proper column for the requested locale.
  - **Option C — translation table**: `project_translations (project_id, locale, title, caption, ...)`; repositories join and resolve translations.
- **Fallback**: if the requested locale is missing, use `defaultLocale` (`en-US`) and then any other meaningful fallback value.

### Practical Rule

- For **UI**, treat this document as a description of current behavior.
- For **domain content**, treat it as architectural direction for the future model.

## Fallback

- **UI (`next-intl`)**: `next-intl` uses `defaultLocale` when a key is missing in the current locale; more fallback behavior can be configured.
- **Domain errors**: the edge should fall back to `en-US` (or `pt-BR`) when the current locale has no entry in `ERROR_MESSAGE`.
- **Domain content** (`LocalizedText`, columns, or translation tables): fall back to `defaultLocale`, then to any available value according to the chosen strategy.

## How To Add Translations

### UI (`next-intl`)

1. Edit `messages/{pt-BR,en-US,es}.json`.
2. Keep the same keys across files, or consciously rely on `defaultLocale` fallback where appropriate.
3. In components, use `useTranslations('Namespace')` and `t('key')`.

### Domain Errors

1. Add the new code to `ERROR_MESSAGE` in `@repo/core` for `pt-BR` and `en-US` (and later for `es` when supported).
2. Ensure edge-level mapping from code → HTTP and code → translated `message` uses the request locale. See [ERROR_HANDLING.md](ERROR_HANDLING.md).

### Domain Content (When Implemented)

- **`LocalizedText`**: fill locale keys such as `{ 'pt-BR', 'en-US', 'es' }` when creating or updating entities.
- **Columns / tables**: fill the corresponding columns or translation rows; on reads, repositories or use cases select the appropriate content for the request locale.
