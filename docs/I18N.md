## Idiomas suportados

| Locale | Uso |
|--------|-----|
| **pt-BR** | Português (Brasil) |
| **en-US** | Inglês (EUA) — `defaultLocale` |
| **es** | Espanhol |

Definidos em `apps/web/src/i18n/routing.ts`:

```ts
locales: ['en-US', 'es', 'pt-BR'],
defaultLocale: 'en-US',
```

## i18n na UI

- **Biblioteca**: [next-intl](https://next-intl-docs.vercel.app)
- **Config**:
  - **Routing**: `src/i18n/routing.ts` — `defineRouting`, `createNavigation` (Link, redirect, usePathname, useRouter)
  - **Server**: `src/i18n/request.ts` — `getRequestConfig`; carrega `messages/{locale}.json`
- **Middleware**: `src/middleware.ts` — `createMiddleware(routing, { localeDetection: true })`; matcher para `/`, `/(en-US|es|pt-BR)/:path*` e exceções (`api`, `_next`, `_vercel`, arquivos estáticos)
- **Mensagens**: `messages/pt-BR.json`, `messages/en-US.json`, `messages/es.json`

### Uso em componentes

```ts
import { useTranslations } from 'next-intl';

const t = useTranslations('Home');
t('projects_title'); // chave sob namespace "Home"
```

## Conteúdo do domínio

### Erros (códigos + mensagens)

- **Core** (`@repo/core`): `ERROR_MESSAGE` em `shared/i18n/ERROR_MESSAGE.ts` — mapa `pt-BR` e `en-US` com códigos estáveis, ex.:
  - `INVALID_DATE_TIME`, `INVALID_ID`, `INVALID_NAME`
- **Regra**: o domínio **não** escolhe o locale; quem chama (web/API) usa o locale do request e consulta um dicionário (pode ser `ERROR_MESSAGE` ou uma camada na borda) para obter a `message`.
- **es**: ainda **não** está em `ERROR_MESSAGE`; ao adicionar, estender `IErrorMessagei18n` em `@repo/utils` e as implementações no Core.

### Conteúdo multilíngue (projetos, posts, etc.)

- **Estado atual**: textos de projetos e experiências estão em uma única língua (ou hardcoded). Não há `LocalizedText` nem colunas por idioma no BD.
- **Plano**:
  - **Opção A — LocalizedText (VO)**: valor como `{ 'pt-BR': string, 'en-US': string, 'es'?: string }`; o use case ou o repositório devolve o texto para o locale pedido (ex.: `?locale=pt-BR`).
  - **Opção B — Colunas por idioma**: `title_pt`, `title_en`, `title_es` (ou `title_pt_br`, etc.); o mapper escolhe a coluna conforme o locale.
  - **Opção C — Tabela de traduções**: `project_translations (project_id, locale, title, caption, …)`; join no repositório.
- **Fallback**: se não houver valor para o locale, usar `defaultLocale` (en-US) e, se ainda faltar, uma string genérica (ex.: `"[sem tradução]"`) ou o primeiro idioma disponível.

---

## Fallback

- **UI (next-intl)**: o next-intl usa o `defaultLocale` quando a chave não existe no locale atual; é possível configurar níveis de fallback.
- **Domínio (erros)**: a borda deve ter fallback para `en-US` (ou `pt-BR`) quando o locale não tiver entrada em `ERROR_MESSAGE`.
- **Conteúdo (LocalizedText/colunas)**: fallback para `defaultLocale` e depois para qualquer valor existente, conforme a opção adotada.

## Como adicionar traduções

### UI (next-intl)

1. Editar `messages/{pt-BR,en-US,es}.json`.
2. Manter as mesmas chaves nos três arquivos (ou aceitar fallback para `defaultLocale` onde faltar).
3. Em componentes: `useTranslations('Namespace')` e `t('chave')`.

### Erros do domínio

1. **Novo código**: adicionar em `ERROR_MESSAGE` em `@repo/core` para `pt-BR` e `en-US` (e `es` quando o tipo for estendido).
2. **Borda**: garantir que o mapeamento código → HTTP e código → `message` use o dicionário e o locale do request. Ver [docs/ERROR_HANDLING.md](ERROR_HANDLING.md).

### Conteúdo de domínio (quando implementado)

- **LocalizedText**: preencher as chaves do locale na estrutura `{ 'pt-BR', 'en-US', 'es' }` ao criar/atualizar entidades.
- **Colunas/tabelas**: preencher as colunas ou linhas correspondentes ao criar/atualizar; na leitura, o repositório ou use case seleciona pelo `locale` do request.
